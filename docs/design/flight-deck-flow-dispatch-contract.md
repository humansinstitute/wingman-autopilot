# Flight Deck Flow Dispatch Contract

Status: proposed design
Last updated: 2026-04-21

## Summary

This document defines how Flight Deck-authored flows should be executed by
Wingmen without moving orchestration intelligence into Flight Deck.

The core decision is:

- Flight Deck owns authored flow definitions and board records.
- Wingmen owns runtime orchestration.
- A flow starts from a normal task record, not a new record family.
- The kickoff task becomes the parent task for the run.
- Wingmen creates all child tasks up front, then advances the run by promoting
  pre-created tasks from `new` to `ready`.

This gives us:

- explicit run structure visible on the board
- deterministic child task descriptions created at run start
- support for fan-out and fan-in through predecessor links
- fast continuation without polling or ad hoc heartbeat cleanup
- a clean place for agent prompts and dispatch capabilities inside Wingmen

## Goals

- Keep Flight Deck as the place where flows are authored and tasks are visible.
- Keep orchestration logic inside Wingmen agent dispatch.
- Materialize all tasks for a flow run at the start of the run.
- Support sequential, parallel, and fan-in flow shapes.
- Make task completion and approval decisions advance the flow through explicit
  dispatch capabilities rather than prompt-only behavior.
- Ensure every agent session can read and write board state through a stable
  local wrapper around Yoke.

## Non-Goals

- We will not create a new `flow_start` record family only to trigger work.
- Making Flight Deck responsible for downstream runtime orchestration.
- Requiring periodic polling to discover the next step.
- Allowing downstream tasks to guess the next task shape after work is already
  underway.

## Current State

Relevant current behaviors:

- Flight Deck `startFlowRun()` only creates the first task of a run.
- Wingmen task dispatch currently activates when a task is assigned to the bot
  and `state === ready`.
- Wingmen does not currently treat predecessor links as a dispatch gate.
- Approval advisories can requeue into an existing flow-run session, but they do
  not yet own durable next-step orchestration.
- The `agent dispatch` UI in Wingmen currently models `chat` and `task`
  dispatch capabilities only.

This design extends the Wingmen dispatch model rather than pushing more logic
into Flight Deck.

## Terminology

### Flow

An authored template in Flight Deck containing ordered steps.

### Kickoff Task

A normal task created in Flight Deck with a `flow_id` attached and
`flow_run_id = null`. This is the trigger for `Flow Dispatch`.

This task is not a disposable launcher. It is converted into the parent task
for the concrete run.

### Parent Task

The kickoff task after `Flow Dispatch` has enriched it with concrete run
context and stamped a `flow_run_id`.

### Child Task

A concrete task created by `Flow Dispatch` for one flow step or one approval
gate. Child tasks share the same `flow_run_id` and point to the parent task via
`parent_task_id`.

### Flow Run

A concrete execution of a flow. It is identified by `flow_run_id`, which is
generated by `Flow Dispatch` and stamped onto the parent task and all child
tasks for that run.

### Dispatch Capability

A distinct Wingmen agent-dispatch mode with its own matcher, prompt contract,
and operator UI surface.

This design introduces:

- `Flow Dispatch`
- `Task Review`
- `Approval Dispatch`

alongside existing:

- `Chat Dispatch`
- `Task Dispatch`

## Record Model

No new record family is required for flow start.

The kickoff signal is a task record with:

- `flow_id != null`
- `flow_run_id == null`
- `state = new`
- assigned to the Wingmen bot

Optional tags should be used to improve readability and diagnostics:

- `flow_kickoff`
- `flow_parent`
- `flow_step`
- `flow_approval`

These tags are not required for correctness if the field matcher is sufficient,
but they are recommended for UX, prompt clarity, and operator tooling.

## Core Model Decision

The kickoff task is the future parent task.

`Flow Dispatch` must not create a second parent task. Its job is to:

1. claim the kickoff task
2. generate a `flow_run_id`
3. convert that kickoff task into the parent task for the run
4. create all child tasks up front
5. set predecessor links on child tasks
6. mark actionable children `ready`
7. leave blocked children `new`
8. exit

This allows the initial task to remain the top-level board object for the run,
while also ensuring every downstream task is articulated up front by the
orchestration agent, not improvised later by worker sessions.

## Dispatch Capabilities

### 1. Flow Dispatch

Purpose:

- instantiate a concrete run from an authored flow
- convert the kickoff task into the parent task
- create child tasks and approval tasks
- stamp a shared `flow_run_id`

Matcher:

- record family is `task`
- assigned to the bot
- `state = new`
- `flow_id != null`
- `flow_run_id == null`

Expected runtime behavior:

1. Patch kickoff task to `in_progress` immediately to claim it and avoid
   duplicate dispatch.
2. Resolve the authored flow from `flow_id`.
3. Generate `flow_run_id`.
4. Patch kickoff task with:
   - `flow_run_id`
   - enriched description of the end goal
   - summary of how the flow will be achieved
   - references to the flow
   - `flow_parent` tag
5. Create child tasks under `parent_task_id = kickoff_task_id`.
6. Number child tasks consistently, for example:
   - `01 - Research Target Market`
   - `02 - Draft Website Outline`
   - `03 - Approval`
7. Stamp each child task with:
   - `flow_id`
   - `flow_run_id`
   - `flow_step`
   - predecessor links
8. Set each child task:
   - `ready` if all predecessors are already satisfied at creation time
   - `new` otherwise
9. Leave the parent task `in_progress` for the lifetime of the run.
10. Stop.

### 2. Task Dispatch

Purpose:

- deliver concrete worker tasks

Matcher:

- existing task-dispatch behavior remains
- assigned to bot
- `state = ready`

This is not orchestration work. It is delivery work.

### 3. Task Review

Purpose:

- inspect a completed worker step
- determine whether any downstream child tasks are now unblocked
- promote all newly-actionable downstream tasks from `new` to `ready`
- activate the next approval task if the next gate is approval

Matcher:

- record family is `task`
- assigned to the bot
- `state = review`
- `flow_run_id != null`

Notes:

- This should be a short-lived orchestration session.
- If a review session already exists for the same task, later updates should be
  routed into that session rather than creating a duplicate session.
- The review step should fan out whenever multiple downstream tasks are now
  unblocked.

### 4. Approval Dispatch

Purpose:

- continue a flow after a human approval decision
- inspect the approval record and linked tasks
- promote downstream tasks from `new` to `ready`
- or mark the run complete if no further work exists

Matcher:

- record family is `approval`
- approval state transitions to `approved`
- `flow_run_id != null`

Notes:

- This is distinct from `Task Review`.
- The approval record, not the approval task reassignment, should be the primary
  machine trigger for continuation.

## `flow_run_id` Lifecycle

This is the key runtime discriminator.

### Before Dispatch

Kickoff task has:

- `flow_id != null`
- `flow_run_id == null`
- `state = new`

This means:

- a flow template has been selected
- no concrete run has been instantiated yet

### During Flow Dispatch

The `Flow Dispatch` session:

- generates `flow_run_id`
- patches the kickoff task with that `flow_run_id`
- creates all child tasks using the same `flow_run_id`

### After Flow Dispatch

All tasks for the run have:

- `flow_id != null`
- `flow_run_id == concrete_run_id`

This ensures:

- the kickoff task no longer matches `Flow Dispatch`
- child tasks can be grouped and routed by run
- review and approval orchestration can reason over the whole run

## Parent Task Lifecycle

Recommended parent task lifecycle:

1. User creates kickoff task in Flight Deck.
   - `state = new`
   - `flow_id != null`
   - `flow_run_id == null`

2. `Flow Dispatch` claims and converts it.
   - patch to `in_progress`
   - stamp `flow_run_id`
   - enrich description
   - keep as parent anchor

3. Parent task remains `in_progress` while child tasks progress.

4. Parent task should only move to a terminal state when the whole run is
   complete.

Open implementation note:

- We should verify whether any existing parent/subtask summary logic in Flight
  Deck tries to derive a parent state from child states. If it does, that logic
  must not fight the explicit parent-orchestration state for flow runs.

## Child Task Creation Rules

Every flow step becomes a child task or approval task.

Rules:

- create all child tasks at run start
- create approval tasks at run start too
- use predecessor links for dependency gates
- support fan-out and fan-in
- preserve `flow_step` identity for traceability

### Actionable State Rules

At creation time:

- if all predecessors are satisfied, set child task `state = ready`
- otherwise set child task `state = new`

At review/approval time:

- when predecessors become satisfied, promote that child task from `new` to
  `ready`

Important:

- downstream tasks must never start merely because they exist
- promotion to `ready` is the runtime gate

## Predecessor Semantics

Predecessor links are used for orchestration, not only display.

This design requires the review/orchestration capabilities to treat a task as
actionable only when all predecessors are satisfied.

Because current task dispatch may still dispatch `ready` tasks even when they
carry predecessor links, the authoritative rule in this design is:

- blocked tasks remain `new`
- only unblocked tasks become `ready`

That allows us to implement this model without immediately rewriting the entire
task-dispatch eligibility system.

## Approval Model

Each approval gate should create:

- an approval record
- an approval task

### Approval Task

Purpose:

- human-visible board object
- gives Pete a task to review
- can carry comments and status history

Lifecycle:

- created up front by `Flow Dispatch`
- starts as `new`
- promoted when it becomes the next gate
- likely moves to `review` or another explicit human-facing state for Pete
- once approved, can be commented and moved to `done`

### Approval Record

Purpose:

- machine-readable gate state
- links to tasks, artifacts, and flow metadata
- authoritative trigger source for post-approval continuation

Lifecycle:

- created up front
- updated by Pete’s decision
- `approved` transition triggers `Approval Dispatch`

### Preferred Continuation Rule

When Pete approves:

1. approval record changes to `approved`
2. optional board comments are posted to linked tasks and approval task
3. approval task may be moved to `done`
4. `Approval Dispatch` runs
5. it promotes next downstream tasks from `new` to `ready`

This is preferred over using approval-task reassignment as the primary machine
trigger.

## Comments And Board Evidence

To keep the board understandable:

- `Flow Dispatch` should leave an execution-plan style comment on the parent
  task summarizing the run structure.
- `Task Review` should leave a concise handoff comment when it unlocks the next
  task set.
- approval decisions should post concise comments onto:
  - the approval task
  - optionally linked child tasks

These comments are for human traceability. They should not be the authoritative
trigger mechanism.

## Wingmen Agent Dispatch UX

The `agent dispatch` surface in Wingmen needs to support more than `chat` and
`task`.

Near-term capabilities:

- `Chat Dispatch`
- `Task Dispatch`
- `Flow Dispatch`
- `Task Review`
- `Approval Dispatch`

Each capability needs:

- separate enable/disable control
- separate prompt template
- separate operator diagnostics
- separate matcher contract

Longer term, this area will need a better UX because the number of dispatch
capabilities will keep growing.

## Stable Board Wrapper

Every session running in any repo must be able to read and write the board.

This should be provided through a stable local wrapper around Wingman Yoke.

Minimum capability set:

- read task
- patch task
- add task comment
- create task
- create approval
- read approval
- read scopes
- read docs
- read chat context where needed

This wrapper should present stable commands regardless of which repo the agent
session started inside.

## `wingman init`

We should add a bootstrap command, likely something like `wingman init`, that
prepares a repo for board-aware execution.

Suggested responsibilities:

1. install or expose the stable local wrapper around Yoke
2. validate auth / connectivity for the current workspace
3. write or update repo-local guidance in:
   - `AGENTS.md`
   - `CLAUDE.md`
4. pin the standard board-work contract for agents in that repo

The goal is that an agent working in any repo on the machine can follow the
same contract for task and approval handoff.

## Prompt Contracts

Each new dispatch capability needs a distinct prompt contract.

### Flow Dispatch Prompt

The prompt should instruct the agent to:

- resolve the flow
- generate `flow_run_id`
- convert kickoff task into parent task
- create all child tasks and approval tasks
- number tasks consistently
- stamp predecessors and references
- mark only currently-unblocked tasks `ready`
- stop

### Task Review Prompt

The prompt should instruct the agent to:

- inspect the reviewed task
- inspect the run state
- determine which downstream tasks now have all predecessors satisfied
- promote those tasks to `ready`
- if the next gate is approval, activate the approval task and ensure approval
  record context is complete
- stop

### Approval Dispatch Prompt

The prompt should instruct the agent to:

- inspect the approval record
- inspect associated tasks and run context
- determine which downstream tasks are now actionable
- promote those tasks to `ready`
- stop

## Example Lifecycle

1. Pete creates a new task in Flight Deck.
   - assigned to Wingman
   - `state = new`
   - `flow_id = website-flow`
   - `flow_run_id = null`

2. Wingmen `Flow Dispatch` matches it.

3. `Flow Dispatch`:
   - patches task to `in_progress`
   - generates `flow_run_id = run-123`
   - enriches task into parent task
   - creates:
     - `01 - Research`
     - `02 - Draft`
     - `03 - Approval`
     - `04 - Publish`
   - marks `01 - Research` as `ready`
   - leaves the rest `new`

4. `Task Dispatch` runs `01 - Research`.

5. Worker finishes and moves `01 - Research` to `review`.

6. `Task Review` matches it, sees `02 - Draft` is now unblocked, promotes that
   task to `ready`, and stops.

7. Later, `Task Review` sees the approval gate is next and promotes the
   approval task.

8. Pete approves. Approval record becomes `approved`.

9. `Approval Dispatch` matches the approval record, promotes `04 - Publish` to
   `ready`, and stops.

10. When the final task completes and no further tasks remain, the parent task
    can be moved to a terminal state.

## Implementation Validation Checklist

This checklist is intended for implementation sign-off, not design review.

### Runtime Validation

- A kickoff task with `flow_id != null` and `flow_run_id == null` triggers
  `Flow Dispatch` exactly once.
- `Flow Dispatch` stamps a `flow_run_id` onto the kickoff task before creating
  downstream tasks.
- The kickoff task becomes the parent task and is not duplicated.
- All child tasks share the same `flow_run_id`.
- Ready tasks begin immediately through normal task dispatch.
- Blocked tasks remain `new`.
- Parallel branches can start together.
- Fan-in tasks do not start until all predecessors are complete.

### Review Validation

- Moving a child task to `review` triggers `Task Review`.
- `Task Review` promotes every newly-unblocked successor, not only one.
- Repeated updates to the same review task do not create uncontrolled duplicate
  orchestration sessions.

### Approval Validation

- Approval task and approval record are created together.
- Approval task does not become active until the run reaches that gate.
- Approval record transition to `approved` triggers `Approval Dispatch`.
- `Approval Dispatch` promotes the correct downstream tasks.
- Approval evidence is visible on the board.

### Bootstrap Validation

- A repo initialized with `wingman init` can read tasks.
- A repo initialized with `wingman init` can patch tasks.
- A repo initialized with `wingman init` can post comments.
- Agents in arbitrary repos can use the same local board wrapper commands.

### Observability Validation

- Operator history clearly shows `Flow Dispatch`, `Task Review`, and
  `Approval Dispatch` outcomes.
- Failures to promote downstream tasks are diagnosable from logs/history.
- Flow-run grouping is visible enough to debug a stuck run.

## Decision Summary

Pinned decisions:

- no new flow-start record family
- kickoff is a normal task with `flow_id` and `flow_run_id = null`
- kickoff task becomes the parent task
- `Flow Dispatch` generates `flow_run_id` and creates all child tasks up front
- blocked tasks remain `new`
- unblocked tasks become `ready`
- `Task Review` promotes newly-unblocked tasks after worker completion
- `Approval Dispatch` promotes newly-unblocked tasks after approval
- approval continuation is triggered by approval-record state change
- tags are acceptable for lightweight task typing
- a stable Yoke-backed local wrapper is required for all repos
- `wingman init` should bootstrap that contract
